# 原作者設計 vs 本組修改 — 對照與說明(報告 / 簡報用)

**日期:** 2026-06-21
**用途:** 整理本組相對原作者(`LeNet_5_Eyeriss_v2`,作者機器 `booo` / `p8101`)所做的修改,供期末報告與簡報呈現。
**範圍:** 聚焦本次除錯討論涉及的三項差異(UART 非同步輸入同步、記憶體實現方式、`clk_de` 時序約束)。

## 項目 1:UART RX 非同步輸入同步(本次核心修正)

**不同之處**
原作者的 `start_en_detect.v`(H2L_detect)只有單一一級 FF,而 `uart_rx_ctl.v` 直接拿外部 USB-UART 的**非同步**訊號 `rx_pin_in` 在 `clk_de` 邊緣採樣——這是一個沒有處理的跨時脈域(CDC)。本組在 `H2L_detect` 內加上標準 **2-FF 同步器**,輸出同步後準位 `pin_sync` 給 `rx_ctl` 與下降緣偵測共用,並對兩顆同步 FF 標 `(* ASYNC_REG = "TRUE" *)`。

**改良原因**
非同步邊緣偶爾落在 FF 的 setup/hold 視窗 → **亞穩態(metastability)** → 偶發 bit 翻轉或 start bit 誤判。由於一張影像要**連續 784 bytes(7,840 bit)全部正確**、且無 ACK / 無重送,任一 byte 被汙染整張就失效。

> **關鍵放大效應(簡報強調點):** 整張成功率 = (單 byte 成功率)^784
> - 單 byte 99.95% → 整張 ≈ 68%(大多成功)
> - 單 byte 99.5% → 整張 ≈ 2%(幾乎全失敗)
> 單 byte 可靠度只差 0.45%,被 784 次方放大成「能用 / 不能用」的懸崖。

`ASYNC_REG="TRUE"` 的作用:強制工具把兩顆 FF 擺相鄰、不被優化掉,最大化亞穩態安定餘裕,使修復不依賴 P&R 擺位運氣。

**改後結果**
重新合成燒錄後**每次傳輸都成功**。反證:只在 UART 輸入端加同步器就 100% 修好,證明根因就是這個未同步的 CDC。

## 項目 2:記憶體實現方式(BRAM → 部分 distributed register)

**不同之處**
原作者絕大多數記憶體用 BRAM 區塊實現;本組因邏輯量較大、資源較吃緊,把約 20% 的記憶體改用 distributed register(LUTRAM,以 fabric 的 LUT/FF 實現)。

**改良原因**
在較大的設計下平衡資源使用(BRAM 與 LUT/FF 之間的取捨)。

**改後結果 / 釐清**
功能正常。需誠實標註的重點:
- 這項改動**不是 UART 間歇失敗的原因**。決定性證據:只加 UART 同步器就 100% 修好;若 LUTRAM 在汙染內部資料路徑,只修輸入端救不回來。
- 但它確實增加 LUT/FF 用量與繞線擁塞,一般可能讓延遲/skew 較差,使本組版本比原作者**更靠近時序邊緣**——這也是為什麼相同性質的 bug 在本組環境更容易現形的「背景因素」之一。
- 因此項目 3(`clk_de` 約束)對本組**更重要**:啟用後 STA 才會真的去檢查這些 LUTRAM 路徑有沒有問題。

---

## 項目 3:`clk_de` 時序約束(讓 STA 真正驗證)

**不同之處**
`clk_de` 是整個設計實際運行的系統時脈(= 125MHz ÷ 8 = 15.625MHz,由 `clock_gen` 的 `clk_count[2]` 除頻而來)。原作者與本組原始狀態都**沒有** `create_generated_clock` 約束。本組已把正確的約束以**註解形式記在 `FPGA_design/constraints/uart/uart.xdc`**,可隨時啟用。

**改良原因**
沒有此約束時,靜態時序分析(STA)不認得 `clk_de` 是時脈,把約 5 萬多個 FF 標成 `no_clock` 並**跳過時序分析**——等於 99.99% 的電路從未被驗證,報告卻顯示「0 failing / 綠燈」。設計能跑只是因為時脈慢(64ns)+ 走低 skew BUFG,餘裕大到沒檢查也剛好過,是「靠運氣」而非「有保證」。

**改後結果**
約束已備妥;啟用重合成後,STA 將涵蓋整個 `clk_de` 域(含 LUTRAM 路徑),把「靠運氣」升級為「有時序保證」。更正規可改用 Clocking Wizard / MMCM 產生 15.625MHz(會自動帶正確時脈約束)。

## 佐證數據(來自原作者 `impl_1` 時序報告,2024-01)

- `no_clock`:**52,596 個暫存器腳位沒有時脈**(根源 `clock_gen_inst/clk_count_reg[2]/Q` = `clk_de`)
- `unconstrained_internal_endpoints`:**154,200 個腳位未約束最大延遲**
- `generated_clocks`:**0 個**
- `no_input_delay`:**2 個輸入埠**(含 `rx_pin_in` 非同步路徑)
- Design Timing Summary:WNS 6.108ns、0 failing,但 **Total Endpoints 只有 3**(只分析到除頻計數器)
- `clk_de` 實體上走 BUFG(`clk_de_BUFG`,帶 52,574 個 FF)→ 低 skew,所以能穩定跑內部邏輯

## 簡報講稿建議(故事線)

1. **問題:** 同一張手寫數字影像,經 USB-UART 送進板子常常要送很多次才成功一次。
2. **排除法:** 查過鮑率(誤差僅 +0.47%,沒問題)、FSM(不會卡死)、IOSTANDARD(已正確)、Python 端(timeout/flush 都有)——全部排除。
3. **真因:** 外部非同步 UART 線直接被系統時脈採樣,沒做 CDC 同步 → 亞穩態;再被「784 bytes 全有全無」放大成間歇失敗。
4. **為什麼原作者能跑、我們不行:** CDC bug 是機率性、對環境敏感;成功率 = 單 byte^784,微小可靠度差異被放大成懸崖。原作者當時運氣好(且設計從未真正收斂時序)。
5. **修法:** 標準 2-FF 同步器 + `ASYNC_REG`;另把 `clk_de` 的 `create_generated_clock` 約束備妥。
6. **結果:** 傳輸 100% 成功。
7. **學到的觀念(加分):** 模擬只驗邏輯、STA 才驗時序餘裕,而兩者都抓不到 CDC/亞穩態——CDC 要靠同步器這種結構手段,這正是數位 IC 面試必考的原因。
