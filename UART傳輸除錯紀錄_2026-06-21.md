# UART 傳輸不穩除錯紀錄

**日期:** 2026-06-21
**平台:** PYNQ-Z2(當作純 FPGA 用,燒 bitstream,非 PYNQ Overlay 流程)
**子系統:** 自製 Verilog UART RX + Windows 端 pyserial 送圖程式
**結論:** 已解決。加上 2-FF 同步器後,每次傳輸都能成功送達板子。

---

## 摘要(TL;DR)

**問題:** PYNQ-Z2 上用自製 Verilog UART 從 PC 經 USB-UART 收手寫數字影像,傳輸不穩——同一張 784-byte 影像常常要送很多次才成功一次,辨識結果經常錯或沒反應。

**根因:** 外部**非同步**的 UART 接收線 `rx_pin_in` 被直接拿到 `clk_de` 邊緣採樣,**沒有做跨時脈域(CDC)同步**。非同步邊緣偶爾落在 FF 的 setup/hold 視窗 → **亞穩態** → 偶發 bit 翻轉或 start bit 誤判。由於一張圖是 784 bytes、無 ACK、無重同步,任一 byte 被汙染整張就失效,於是低機率的單 bit 錯誤被放大成「要傳很多次」。**不是鮑率、不是接線、不是 IOSTANDARD**(這些都查過、排除了)。

**解法:** 在 `H2L_detect` 內加上**標準 2-FF 同步器**,讓 start bit 偵測與資料採樣都改吃同步後的乾淨訊號。重新合成 → 實作 → 燒錄後,**每次傳輸都成功**。

**一句話:** 這是一個教科書等級的 CDC 未同步 bug——非同步輸入沒過同步器。

---

## 名詞快速解釋(先讀這個)

**`clk_de`** — 整個設計實際運行的系統時脈,**不是**板子進來的 125MHz 原始時脈。`clock_gen.v` 把 125MHz 除以 8 得到 **15.625MHz**(`clk_de = clk_count[2]`)。除了那顆小小的除頻計數器跑在 125MHz,其餘 5 萬多個暫存器(加速器 + UART)全吃 `clk_de`;UART 鮑率分頻也是以 15.625MHz 為基準算的。名字可理解成「放慢/除頻版的 clock」。

**STA(Static Timing Analysis,靜態時序分析)** — Vivado 在 P&R 後,不靠模擬、純用路徑延遲檢查「每條暫存器→暫存器的資料有沒有在一個時脈週期內穩定到達(setup),又沒早到破壞(hold)」的把關步驟。它必須先知道「哪條線是時脈、週期多少」(= 時脈約束)才能檢查。**這次事件中 STA 本該把關卻被蒙眼:** clk_de 沒約束 → 被當 `no_clock` 跳過 → 綠燈放行未驗證的設計;而且即使有檢查,STA 本質上也抓不到 CDC/亞穩態(那要靠同步器 + ASYNC_REG 與 CDC lint),屬雙重失守。

**skew(時脈偏斜)** — 同一個時脈邊緣抵達不同暫存器的時間差。理想上所有 FF 同時收到邊緣;**low skew = 幾乎同時到達**,靠走專用全域時脈網路(BUFG → 平衡的 clock tree)達成。skew 太大會造成 hold violation(資料互相蓋掉),且不會因時脈變慢而改善。

**BUFG** — FPGA 內的全域時脈緩衝器,把訊號送上低 skew 的專用時脈樹。本案 `clk_de` 確實有走 BUFG,所以實體時脈品質其實是好的——缺的只是「時序約束」這層驗證。

## 為什麼 CDC 這麼重要(也是數位 IC 設計面試幾乎必考的題目)

這次的 bug 不是個案,而是 **CDC(Clock Domain Crossing,跨時脈域)** 這一整類問題的典型代表。值得把它的份量講清楚,因為面試常考,而這次剛好親手踩到、親手修好,是很好的實戰素材。

**CDC 在真實晶片裡無可避免。** 任何稍具規模的 SoC 都有多個時脈域:CPU、各週邊、記憶體介面、不同 PLL 出來的時脈、還有像本案這種「外部非同步訊號進晶片」的情形。訊號只要從一個域跨到另一個不相關的域,就是一次 CDC,設計者**一定**得處理。

**它危險,是因為會產生亞穩態(metastability)。** 當一顆 FF 取樣到一個違反 setup/hold 的輸入,它的輸出可能在一段**不確定的時間**內卡在 0 與 1 之間,再隨機收斂。這個壞值若直接傳給下游,就造成功能錯誤。亞穩態**無法完全消除**,只能用同步器把發生機率壓到極低(以 MTBF 量化,可長到數十年一次)。

**面試為什麼幾乎都問這題,有幾個很實際的理由:**

1. **模擬抓不到。** RTL 模擬是零延遲、cycle-accurate 的,根本不模擬亞穩態——所以 CDC bug 在模擬裡常常「完全正常」,卻在真晶片上間歇發作(就像本案:行為對、波形對,板子上卻時好時壞)。這種「模擬過了、矽片卻爆」的 bug 最貴、最難 debug,公司非常在意你懂不懂。
2. **它區分「只會寫 Verilog 語法」和「真的懂硬體時序」的人。** 能不能意識到「這條訊號跨域了、要同步」,直接反映候選人有沒有真實硬體經驗。
3. **它有明確的標準解,適合層層追問深度。** 從最基本一路問下去:
   - **單 bit 控制訊號** → 2-FF(有時 3-FF)同步器。
   - **多 bit 匯流排不能只用 2-FF 同步**(各 bit 會在不同週期穩定下來,組合出根本沒出現過的錯值)→ 要用 **Gray code**(給計數器/指標)、**非同步 FIFO**,或 **req/ack 握手**。
   - **脈衝跨域**(尤其快→慢)可能整個被漏掉 → 用 **toggle synchronizer**。
   - 同步器兩級之間**不能插組合邏輯**;要注意 reconvergence、訊號相關性等細節。

**用本案當面試答題範例:** 「我做 FPGA UART 時遇到傳輸間歇失敗。查過鮑率(誤差僅 0.47%)、FSM、IOSTANDARD 都沒問題。最後發現是外部非同步的 UART RX 線直接被系統時脈採樣,沒過同步器 → 亞穩態造成偶發 bit 翻轉;又因為一次傳 784 bytes 且無重送機制,任一 byte 壞掉整包失效,所以表現成『要傳很多次』。解法是加標準 2-FF 同步器,之後 100% 成功。」——這個故事同時展示了 debug 流程(排除法)、對亞穩態/MTBF 的理解、以及對『模擬過≠矽片對』的體會,正是面試官想聽的。

## 1. 問題現象

在板子上跑 LeNet-5 / Eyeriss-v2 加速器 demo 時,從 PC 經 USB-UART 把手寫數字影像送進 FPGA,**通訊不穩定:同一張圖往往要送很多次才會成功一次**,辨識結果(板子上 LED / 七段顯示器)經常是錯的或沒反應。

關鍵背景:host 端每次送的是一張 **28×28 = 784 bytes** 的影像,FPGA 要**連續收滿 784 bytes(= 7,840 個 bit)全部正確**才會觸發推論。這個「全有全無」的特性是後面分析的重點。

## 2. 系統架構(本次相關部分)

```
PC (pyserial, 115200 8N1)
   │  USB-UART adapter (非同步, 3.3V LVCMOS33)
   ▼  rx_pin_in (PYNQ-Z2 pin W6)
TOP_integration_uart.v
   ├─ clock_gen        : clk(125MHz) ÷8 → clk_de(15.625MHz)  ← 全設計時脈
   ├─ uart_module
   │    └─ uart_rx_module
   │         ├─ H2L_detect (start_en_detect.v)  下降緣偵測
   │         ├─ uart_rx_band_gen                鮑率產生 (1 tick / bit)
   │         └─ uart_rx_ctl                     接收 FSM, 寫進 ifmap BRAM
   ├─ interrupt_gen    : 收到資料後固定逾時 → 觸發 reset
   └─ TOP_interface    : 加速器本體
```

- 這版 bitstream 是 **RX-only**(`uart.xdc` 只約束了 `rx_pin_in`,沒有 TX 腳位);結果只顯示在板子上,不回傳 PC。
- Host 程式:`host_demo/fpga_uart_demo.py`,tkinter 畫布手寫 → 縮 28×28 → 一次 `write()` 784 bytes。

## 3. 排查過程與被排除的假設

一開始的幾份除錯指南把矛頭指向「鮑率不準」「FSM 卡死」「IOSTANDARD 漏寫」。實際讀程式碼後,**這些全部被證據排除:**

| 假設 | 結論 | 證據 |
|---|---|---|
| 鮑率分頻算錯 / 時脈不一致 | ❌ 鮑率其實是對的 | `uart.xdc:4` clock 125MHz;`clock_gen.v:8` `clk_de=clk_count[2]`=125/8=15.625MHz;`uart_rx_band_gen.v:10` `SYS_RATE=15.625MHz` 兩者吻合。`CNT_BAND=15625000/115200≈135` → 實際 ≈115,740 baud,對 115200 **誤差僅 +0.47%**,遠在容忍範圍內。 |
| RX 用 16 倍過採樣,公式 clk/(baud×16) | ❌ 本設計非 16 倍過採樣 | `uart_rx_band_gen.v` 每 bit 只產生 1 個 tick;置中採樣靠 `cnt_bps<=HALF_CNT_BAND` 的半 bit 起始偏移達成。 |
| FSM 卡在 STOP bit 出不來,要重插拔 | ❌ FSM 不會 deadlock | `uart_rx_ctl.v:35` END 狀態無條件 `END→BFREE→IDLE`,永遠回到 IDLE。 |
| IOSTANDARD 漏寫導致訊號抖動 | ❌ 已正確設定 | `uart.xdc:18` `rx_pin_in`(W6)與 clock 都標了 `LVCMOS33`。 |
| Python 缺 timeout/flush 會丟字 | ❌ host 端沒問題 | `fpga_uart_demo.py:76,112,114` 有 `timeout=1`、`reset_output_buffer()`、`flush()`。 |

## 4. 真正的根本原因:CDC(跨時脈域)未同步

`rx_pin_in` 是來自外部 USB-UART adapter 的**非同步訊號**——由 PC 端自己的振盪器驅動,與 FPGA 的 `clk_de` 沒有任何時脈關係,邊緣到達時間完全隨機。

問題在於設計**直接拿這條非同步訊號在 `clk_de` 邊緣採樣**,沒有經過同步器:

- `start_en_detect.v`(H2L_detect)原本只有**單一一級 FF**,且 `sig_H2L` 用原始 `pin_in` 組合邏輯運算。
- `uart_rx_ctl.v:76-83` 更是**直接採樣原始 `rx_pin_in`** 寫進 `rx_data`。

當訊號邊緣剛好落在 FF 的 setup/hold 視窗內,FF 會進入**亞穩態(metastability)**,輸出短暫卡在 0/1 之間並可能解析成錯值 → 偶發 bit 翻轉。

**為什麼是「偶爾才成功」而不是「完全不通」:** 每個 bit 出錯機率很低,但一張圖要連續 7,840 個 bit 全對才算成功;極低的單 bit 錯誤率累乘起來,就變成整張圖經常壞、偶爾才全對。這個「784-byte 全有全無、無 ACK / 無重同步」的架構,把任何微小錯誤放大成了「要傳很多次」。

### 修改前為何失敗、修改後為何成功(因果對照)

**先澄清:** 這裡不是「漏收 / 丟字」(不是 FIFO 被 reset 那種),而是收到的資料被**汙染或位元錯位**,使整張影像作廢。

**修改前 — 為什麼一次傳輸會失效:**
非同步的 `rx_pin_in` 邊緣偶爾剛好落在 FF 的 setup/hold 視窗內,FF 進入亞穩態。亞穩態會造成兩種傷害:

1. **資料 bit 採錯** — `uart_rx_ctl.v` 直接採原始腳位寫進 `rx_data`,某個 bit 解析成錯值 → 該 byte(該像素)內容錯。
2. **起始位元偵測亂掉(更嚴重)** — H2L_detect 對雜訊/抖動誤判出**假的下降緣**,或**漏掉/延後**真正的 start bit → 該 byte 的位元邊界整個錯位,後面的 bit 跟著一起歪。

由於一張圖是 **784 bytes 連續、無 ACK、無重同步**,只要其中**任何一個 byte** 被汙染或錯位,整張影像就失效 → 推論結果錯 → 對使用者就是「這次沒成功,要再送」。單 bit 出錯機率雖低,但要 7,840 個 bit 全對才算贏,累乘下來自然經常輸、偶爾才贏。

**修改後 — 為什麼每次都成功:**
2-FF 同步器把非同步訊號「馴服」成乾淨的同步訊號:

1. **第 1 級** 先把 `rx_pin_in` 打進 `clk_de` 域;就算這一拍亞穩,也有**一整個 clk 週期**讓它自然收斂到穩定的 0 或 1。
2. **第 2 級** 再採一次,輸出幾乎必為**穩定、與 clk_de 對齊、無毛刺**的準位。

之後 start bit 偵測與每一個資料 bit 看到的都是穩定值 → 不再有隨機翻轉,也不再有假起始邊緣 → 784 bytes 的每個 bit 都正確 → 整張影像完整無誤 → **每次傳輸都成功**。換句話說,真正改變的不是「速度」或「鮑率」,而是把出錯機率從「每張圖夠高到經常壞」壓到「可忽略」。

### 次要風險(本次未改,列為後續)
- **`clk_de` 是用計數器 bit 當時脈**(`clock_gen.v:8`),走一般 fabric 繞線(非 BUFG/MMCM),且 XDC 沒有 `create_generated_clock` → 這個 15.625MHz 域等於未被時序分析,可能有 setup/hold 邊緣違規。
- **`interrupt_gen` 固定逾時**(`interrupt_gen.v:13`,2,000,000 @ 15.625MHz ≈ 128ms)從第一個下降緣起算後 reset 整條 datapath;784 bytes 傳輸約 68ms,餘裕 ~60ms,平常夠但脆弱,host 啟動慢或 USB 分批時可能把 reset 打在收圖中途。

## 5. 採用的修法

在 **H2L_detect 內部加 2 級同步器**,把同步後的準位輸出,讓 H2L 偵測與 rx_ctl 都改吃同步訊號。第 1 級就算進亞穩態,給它一個完整週期安定;第 2 級輸出幾乎必為穩定 0/1,把出錯機率壓到可忽略。

### 改動檔案 1:`FPGA_design/src/UART/start_en_detect.v`
- 新增 `sync_0 → sync_1` 兩級同步暫存器,輸出新 port `pin_sync`(同步後準位)。
- `sig_H2L` 改用 `sync_1` 與其前一拍 `pin_pre` 做下降緣偵測(全在同步域)。
- reset 時三個暫存器都設為 `1'b1`(UART idle 為高),避免假邊緣。
- 副作用(正向):`interrupt_gen` 內的 H2L_detect 也一併變安全(新 output 懸空,Verilog 合法,不需改 interrupt_gen)。
- **(2026-06-21 補)** 同步暫存器標上 `(* ASYNC_REG = "TRUE" *)`:強制工具把兩顆 FF 擺相鄰、不被優化掉,最大化亞穩態安定餘裕,讓修復不再依賴 P&R 擺位運氣(避免未來重新合成時 robustness 又掉回邊緣)。

### 改動檔案 2:`FPGA_design/src/UART/RX/uart_rx_module.v`
- 新增 `wire rx_pin_sync`,接到 `H2L_detect.pin_sync`。
- 把 `uart_rx_ctl.rx_pin_in` 從原始 `rx_pin_in` 改接 `rx_pin_sync`。

> 代價:多 2 個 clk 延遲。在 15.625MHz(週期 64ns)對 115200 baud(bit 週期 8.68µs)完全可忽略。鮑率、FSM、BRAM 邏輯均未更動。

## 6. 結果

重新**合成 → 實作 → 產生新 bitstream → 燒錄**後測試:**每次傳輸都成功送達板子,不再需要重送。** 確認根因為非同步輸入未做 CDC 同步。

(提醒:改的是 RTL 原始碼,板子跑的是編譯後的 `.bit`,必須重跑合成/實作流程才會生效——不是存檔或重載就好。)

## 7. 為什麼原作者的 .bit 以前能跑、現在卻幾乎收不到?

這是這次最值得記下來的觀念。徵狀:**很久以前直接燒原作者給的 .bit,大多數傳輸都吃得到;現在燒同一個 .bit(尚未加 2-FF)卻幾乎吃不到。** 同一份 bitstream、同一份 RTL,行為卻天差地別——這正是 CDC bug 的典型樣貌。

### 7.1 從原作者 Vivado 專案翻到的鑑識證據(鐵證)

路徑:`FPGA_design/Vivado/PYNQ_Z2/`

- **原作者的設計從頭就沒有同步器。** `.xpr` 用相對路徑 `$PPRDIR/../../src/UART/start_en_detect.v` 參考原始碼,也就是本次編輯的同一批檔。所以原作者建 .bit 時用的就是「沒有 2-FF 同步器」的 RTL——**原作者的 .bit 和早期燒的是同一份有潛在 bug 的電路。**
- **整個設計從來沒有真正收斂時序。** `impl_1/..._timing_summary_routed.rpt`(2024-01-11,原作者機器 `booo` / `C:/Users/p8101/...`)顯示:
  - `no_clock`:**52,596 個暫存器腳位沒有時脈**,根源是 `clock_gen_inst/clk_count_reg[2]/Q`(即 `clk_de`)。整個加速器 + UART 邏輯所在的時脈域,Vivado 根本不認得它是時脈,沒做任何時序分析。
  - `unconstrained_internal_endpoints`:**154,200 個腳位未約束最大延遲(HIGH)**。
  - `generated_clocks`:**0 個**(證實 XDC 沒對 `clk_de` 下 `create_generated_clock`)。
  - `no_input_delay`:**2 個輸入埠沒 input delay**(`rx_pin_in` 的非同步路徑完全沒被檢查)。
  - Design Timing Summary:WNS 6.108ns、0 failing,但 **Total Endpoints 只有 3 個**——報告只分析了那顆除頻計數器,功能電路一個都沒算到。

結論:原作者大概看到「0 failing / 綠燈」就以為時序過了,但**幾乎整片電路根本沒被分析**。這顆晶片一直是「靠運氣跑起來」,不是「設計保證會跑」。

### 7.2 為什麼「以前穩、現在不穩」——784 次方的放大效應

一張圖要 **784 bytes 全部正確**才算成功,因此:

> **整張成功率 = (單一 byte 成功率)^784**

這個次方對單 byte 可靠度極度敏感:

| 單 byte 成功率 | 每 byte 錯誤率 | 整張成功率 (^784) | 體感 |
|---|---|---|---|
| 99.95% | 0.05% | ≈ 68% | 大多吃得到 |
| 99.9%  | 0.1%  | ≈ 46% | 時好時壞 |
| 99.5%  | 0.5%  | ≈ 2%  | 幾乎吃不到 |

單 byte 物理可靠度只要從 99.95% 掉到 99.5%(實際只差 0.45%,極小的變化),使用者體感就從「大多成功」**斷崖式**掉到「幾乎全滅」。784 次方把微小的可靠度差異放大成了「能用 / 不能用」的懸崖。

### 7.3 那「以前那次」為什麼單 byte 可靠度比較高?

因為沒有同步器的設計**沒有 margin 保證**,單 byte 可靠度完全取決於當下外部條件:

- 用哪顆 USB-UART adapter(晶片不同 → 實際鮑率誤差、驅動能力、上升沿不同)
- 線材長度 / 品質、有沒有共地、接觸品質
- 主機 PC、USB 驅動 / latency timer 如何切 byte(影響 byte 間相位重對齊)
- 非同步邊緣與 `clk_de` 的隨機相位、溫度、電壓
- 連 P&R 延遲都是「未約束時脈域下工具隨便擺的結果」

當時這些剛好落在「幸運側」→ 單 byte 可靠度高 → 整張大多過。現在換了環境(adapter / 線 / PC / 那天溫度),把單 byte 可靠度往下推了一點點,經 784 次方放大就變成幾乎全滅。**加上 2-FF 同步器後 100% 成功,反證了主導因素就是這個沒被同步的取樣,而非鮑率或接線。** 這就是教科書說 CDC bug 危險的原因:同一個 bitstream 在 A 環境正常、B 環境壞掉,難重現、難 debug。「原作者能跑」不代表設計是對的,只代表他當時運氣好。

## 8. 延伸問答(深入觀念釐清)

**Q1. 單 byte 可靠度是在合成、實作、還是燒板時固定?**
分兩半。合成決定「有哪些 FF / 邏輯」;**實作(P&R)固定內部延遲、擺位、skew**=「內部那一半 margin」,同一個 .bit 就一樣;**runtime 才產生外部那一半**:非同步邊緣與 clk_de 的隨機相位、溫度、電壓、adapter 鮑率、訊號完整性,每次跑都不同。而且亞穩態本質上是 runtime 機率事件——.bit 固定的是「出錯機率」,不是「會不會出錯」。

**Q2. 設備一樣、中間重開機過一次,會有影響嗎?**
可能有。重開機會重置主機 USB/serial 驅動與 latency timer、重新列舉 adapter、重置熱狀態、並重新隨機化開機相位。對一個沒有 margin 保證的設計,任一項都可能把單 byte 可靠度推一點點,再被 784 次方放大(見 §7.2)。但重點是:有同步器 + 有約束的設計根本不該在意重開機;會因重開機就時好時壞,本身就是「靠運氣」的徵兆。

**Q3. 沒有 generated clock 約束,那些 FF 吃哪來的 clock?可以嗎?怎麼還能跑?**
「沒有 generated-clock 約束」≠「沒有時脈訊號」。實體上 `clk_de` 是一條走 BUFG 的真實全域時脈(clock_utilization 報告:`clk_de_BUFG` 帶 52,574 個 FF),FF 照常被 clock。缺的只是**時序約束**——告訴 STA「這是 15.625MHz 時脈,請檢查」。沒有它,STA 把那些 FF 標成 `no_clock` 並跳過分析。能跑,是因為時脈真實 + 慢(64ns 週期)+ 低 skew(BUFG)→ 路徑用很大餘裕滿足 setup/hold,沒人檢查也剛好都過。風險是:萬一某路徑真的太長,工具不會警告;非同步輸入也沒被分析。**能跑 ≠ 被驗證過。**

**Q4. 把約 20% 記憶體從 BRAM 改成 distributed register(LUTRAM),跟更容易失敗有關嗎?**
證據指向無關。**決定性證據:只在 UART 輸入端加一個同步器,成功率就回 100%。** 若 20% LUTRAM 在汙染內部資料路徑(setup/hold),只修 UART 輸入救不回來。LUTRAM 確實會增加 LUT/FF 用量與擁塞、一般可能讓延遲/skew 變差,但本設計慢(15.625MHz)又有低 skew BUFG,內部餘裕大,頂多讓你的版本比原作者更靠近邊緣,不是失敗主因。提醒:磁碟上的時序報告是原作者 BRAM-heavy 版本(機器 `booo`,2024-01);你 LUTRAM 版本的利用率/skew 要看你自己重合成後的報告,加上 `create_generated_clock` 後 STA 就會幫你檢查那些路徑。

**Q5. 為什麼 `clk_de` 要用原始時脈的 1/8,而不直接用 125MHz?**
主因是**時序收斂**。這顆加速器(Eyeriss-v2 PE 陣列、GEMM、位址產生)的組合邏輯關鍵路徑很長,在 125MHz(8ns 週期)內算不完 → 會 setup 違規。降到 15.625MHz(64ns)等於每條路徑多 8 倍時間,長運算路徑就輕鬆滿足。白話:設計本身跑不到 125MHz,所以把現成的快時脈除頻成跑得動的速度。附帶好處:全設計維持單一時脈域(避免內部再生 CDC)、慢時脈較省電、計數器除頻最好寫。125MHz 是 PYNQ-Z2(H16 腳)現成提供的時脈。**註:** 用計數器 bit 除頻是省事但粗糙的做法,正規應改用 Clocking Wizard/MMCM,它會自動帶正確的時脈約束,就不會發生本案「沒 generated clock」的問題。

**Q6. 「模擬(simulation)」和 STA 差在哪?為什麼還需要 STA?**
**功能模擬(RTL simulation / testbench)只檢查「邏輯對不對」**——給定輸入,輸出波形是不是設計者想要的值。它用理想化時脈、訊號在一個週期內視為瞬間傳遞,所以**完全不知道每條路徑在真實晶片上的延遲,也不知道每顆 FF 還剩多少 time margin**。**STA 則完全不用測資**:拿 P&R 後帶真實閘 + 繞線延遲的網表,對**每一條路徑**算最壞情況的到達 vs 需求時間 → 得到 setup/hold 餘裕(slack),涵蓋所有路徑與所有溫壓角落。兩者互補:模擬保證「邏輯正確」,STA 保證「時序正確、還有多少餘裕」。
這也解釋了為什麼本案兩道防線都沒攔到:RTL 模擬看到的波形是對的(邏輯沒錯),STA 又因沒約束而沒分析那個時脈域;而且**就算 STA 有分析,它也驗不出 CDC/亞穩態**(STA 與模擬都用理想時脈、不模擬亞穩態),CDC 只能靠同步器這類結構手段 + CDC lint / 設計審查抓。所以它只在真硬體上、間歇地現形。
